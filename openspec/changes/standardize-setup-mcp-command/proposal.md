## Why

`gitnexus setup` currently persists resolved binary paths such as `/usr/local/bin/gitnexus` into generated MCP client configuration. Users want setup-managed MCP entries to be portable, readable, and consistent by invoking GitNexus as `gitnexus mcp` everywhere instead of storing absolute paths or package-manager fallback commands.

## What Changes

- Standardize all setup-generated MCP entries to invoke GitNexus through the command name `gitnexus` with `mcp` as the server argument.
- Update Cursor, Claude Code, OpenCode, Codex, Gemini CLI, and any other setup-managed MCP config writers to avoid persisting resolved absolute binary paths.
- Remove or bypass package-manager fallback MCP config generation such as `npx -y gitnexus@latest mcp` for setup-managed entries.
- Keep setup detection and reporting useful, but ensure detection results do not alter the MCP command written to config.
- Update tests and docs to assert the normalized `gitnexus mcp` command across supported clients.

## Capabilities

### New Capabilities
- `setup-mcp-command`: Defines the normalized command contract for MCP entries generated by `gitnexus setup`.

### Modified Capabilities
- None.

## Impact

- Affected code: `gitnexus/src/cli/setup.ts` and setup helpers that construct MCP entries for supported clients.
- Affected tests: setup JSONC/TOML integration and unit tests that currently expect absolute paths or fallback commands.
- Affected systems: user MCP config files for Cursor, Claude Code, OpenCode, Codex, Gemini CLI, and related setup-managed clients.
- User impact: rerunning `gitnexus setup` rewrites GitNexus MCP entries to call `gitnexus mcp`; editor processes must have `gitnexus` resolvable on PATH.
